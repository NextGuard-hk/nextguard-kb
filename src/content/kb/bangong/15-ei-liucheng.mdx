# EI相关流程

## 工单/EI流程分类

新EI流程分为3个类别，主要是前2个：

### 1. BUG类

常规流程：SE开工单 > TS分析/验证 > 确认EI > 研发修复/升级EI > 必要的话出HF > 关闭EI

### 2. 新功能/改进类

常规流程：SE直接开EI > TS协助确认 > PM分析 > 创建GAT > 关闭EI

### 3. Backport类

常规流程：SE直接开EI > SED确认 > Mike确认 > Saipu确认 > QA验证 > TS发布 > 关闭EI

## EI处理流程与职责

### BUG/问题处理流程

1. EI由TS分析/验证
2. 确认后转研发协助
3. 研发提供解决方案或Hotfix
4. QA验证后由TS发布Hotfix

### 新功能/改进流程

1. EI由SE直接提交
2. TS协助确认是否为新功能
3. 确定是新功能后PM协助分析
4. 有解决方案后创建GAT

## EI跟踪响应规范

- **New EI**: 需要研发支持，尽快转给相关的研发Manager，不要放到自己手里
- **EI在研发手里**: 等待研发回复，至少每1-3天询问研发最新进展情况，比较着急且长时间未响应的，发邮件通知Manager
- **EI在自己手里**: 研发等待回复，至少1-3天对EI进行update，不要放在自己手里
- **重点客户**: 针对Top用户的EI或Case每天总结最新状态通知到Manager

## EI关闭解决类型 (5选1)

| 类型 | 说明 | 是否需要KB |
|------|------|----------|
| 无（默认） | - | - |
| Hotfix | 通过热修复解决 | - |
| New-version | 通过新版本解决 | - |
| Configuration | 通过配置解决或研发提供的方法解决 | 需要KB |
| TS-Internal | 根据TS总结的经验解决，不需要研发协助 | 需要KB |
| By-Design | 产品设计如此，不需要修改 | - |

## EI提交基本信息要求

目标不是为了设置一个阻碍，而是能够准确的、判断和解决问题。

EI前需考虑的事情：
- 有没有搜索过相关的KB？
- 有没有搜索过相关的Case？
- 有没有内部沟通讨论过？
- 目前所做的事情，分析的结果，背后的场景是否体现在了EI上？
- 换位思考，如果你是研发看到了这个EI能否清楚了解故障现象及希望实现的功能？
- EI的有效信息是否有提供给研发？

## 问题与建议

建议：TS把快速通道先建立起来，通过微信或其他工具，让研发协助我们跟用户解释清楚某个操作的原因，把事情进行下去，结束之后写总结的时候，和研发一起把内容整理成KB分享出来。但有个前提，不要问研发重复的问题，也就是说此问题是第一次出现。
