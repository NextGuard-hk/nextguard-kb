# KB-高级邮件路由相关问题

## 案例1：UCSS平台系统日志频繁报连接下一跳邮件路由失败

### 背景

某客户反馈系统日志反复出现某一台邮件dlp报警，提示连接邮件下一跳邮件路由失败，且该报警信息基本5-10分钟一次反复的恢复然后再次触发。

客户环境有ASEG-A、ASEG-B两台设备，MTA配置均配的是同一个邮件网关地址，但是只有DLP-A有报警日志。

### 原因分析

1、查看邮件日志，邮件基本都正常投递了，一天只有1-2封邮件出现投递到下一跳邮件网关回复连接超限被拒、超时投递的现象，频率和页面上每5-10分钟一次的告警相比很低，没有充分证据证明是下一跳邮件路由引起的问题。

ASEG-A、ASEG-B的配置一样，MTA配置默认路由都是同一个地址。

2、ASEG中有个检测脚本，5分钟检测一次，检测下一跳邮件路由30s没有响应的话，就会报连接下一跳邮件路由失败的告警，检测脚本如下：

```bash
vi /usr/libexec/postfix/mta_alert.sh
```

```bash
`perl -e "alarm 30; exec @ARGV" "nc -zv $_host $_port 2>/dev/null 1>/dev/null"` 2>/dev/null 1>/dev/null
```

可以看到检测脚本实际上执行的是`nc -zv $_host $_port`的端口检测命令，我们用这个命令在ASEG-A、ASEG-B上分别测试联通时间，应该可以找到问题的根源。

3、分别在ASEG-A、ASEG-B上执行NC端口检测耗时对比：

明显ASEG-A这台连接测试用时在30s左右，基本就是在告警红线的左右徘徊，但是ASEG-B基本保持在20s以内，所以一直没有告警。

4、现在基本可以判断是ASEG-A上检测命令超时导致，但是为什么A超时，B不超时呢，同样的局域网、同样的下一跳路由。

如果仔细观察截图会发现，端口测试时有一行报错："172.30.3.26: inverse host lookup failed: host name lookup failure"

nc命令的"-v"参数，是一个反向域名解析，那么超时原因会不会是因为dns对域名解析超时导致呢？

5、取消参数-v后，再次测试，基本断定就是客户dns服务器对邮件路由ip的域名解析失败耗时导致。正常情况下基本解析失败也是ms级别返回，但是客户环境却需要20s以上。

6、ASEG-A配置了两个dns，导致nc会去这两个dns上去查域名，结果两个dns都返回失败，耗时更长。

最后抓包验证dns回复一次失败需要5s，确实是客户环境dns服务回复解析结果太慢导致，由于dns查询是udp，基本判断故障出现在网络或者dns服务器那边。

### 解决办法

nc这个命令的参数"-v"暴露了客户环境dns对172.30.3.26（邮件网关）的域名解析超时的问题。

既然我们的目的只是检测邮件网关的25端口是否可达，-v参数其实检测意义就不大了，我们系统也不关心域名是否存在或者是否可达，所以推荐的优化方法就是去掉检测命令中的参数-v。

```bash
vi /usr/libexec/postfix/mta_alert.sh
```

将`nc -zv`改为`nc -z`即可。

### 备注

平台报"连接邮件下一跳邮件路由失败"，系统是5分钟检测一次，30s内端口无响应就会报警。

测试方法：

```bash
/usr/libexec/postfix/mta_alert.sh  # 报警脚本路径
perl -e "alarm 30; exec @ARGV" "nc -zv 172.21.23.88 25"
```

这行命令意思就是执行一个命令，30秒没结果，直接alarm。

```bash
echo $?
```

如果输出142，表示错误码。

### 修改默认30s方法

将alarm后的30改为60，不要改太大，60最多。

如果碰到dns解析超时，去掉脚本-v即可（-v是域名反向解析命令）。

### 总结

1、不是检测命令有问题，只是命令中-v的参数暴露了客户环境dns对邮件网关IP地址的域名解析很慢的问题（正常环境dns回复超时失败也就ms级别，但是客户环境要15-25s）。

2、既然目的只是检测邮件网关的25端口是否可达，-v参数其实检测意义不大，推荐的优化方法就是去掉检测命令中的参数-v。

### 深层次原因分析

- 端口检测命令带了参数"-v"，dsg的健康检查下一跳端口是否正常，用的脚本是：`nc -zv $_host $_port`
- 每五分钟执行一次`nc -zv 172.30.3.26 25`的端口检测，如果30s内端口检测不通，系统就会认为下一跳不可达
- nc命令的"-v"参数会触发域名反向解析，去linux配置的dns服务器上查这个ip对应的域名
- 客户dns服务器返回域名查询失败导致超时
- 抓包测试显示从nc -v命令开始执行到dns服务器返回查询失败需要三个回合
- DLP-B配置了一个dns，测试超时返回大概是18-25s，小于30s告警阈值
- DLP-A配置了2个dns，两个超时加起来基本是25-35s区间，容易触发30s告警阈值
