# UCSS审批通知常见案例

## 案例1、手动发送邮件审批通知方法

根据网络事件id，手动发送邮件审批通知。

**注：针对3.3版本，只适合sps-109之前的版本**

3.3版本需要进入redis里执行以下两个命令：

```bash
zRemRangeByScore network_incident_id_ser
hset network_incident_id_list_key_temp_m
```

### 3.3打sps-109后，手动发送通知方法

策略通知主队列：

```
network_incident_notify_list
endpoint_incident_notify_list
mobile_incident_notify_list
```

策略通知重试队列：

```
network_incident_notify_list_retry
endpoint_incident_notify_list_retry
mobile_incident_notify_list_retry
```

redis-cli执行命令（手动重投）：

```bash
RPUSH network_incident_notify_list '{"type":1,"id":1,"third":false,"created":...}'
RPUSH network_incident_notify_list_retry '{"type":1,"id":1,"third":false,"created":...}'
```

id：就是要重投的网络事件id

### 审批通知发送失败脚本操作流程

一、提取审批通知发送失败的网络事件id：

```bash
cd /opt/skyguard/sps/logs/sps-incident
zcat incident.2023-0* |grep 'email send failed'
cat incident.log |grep 'email send failed'
cat list | awk -F '--' '{print $1}'
```

二、手动操作重发邮件通知：

```bash
vim sendMailNotification.sh
```

## 案例2、foxmail等客户端邮件审批回复显示乱码

其实不是乱码，是正常的URL编码。

解决方法：浏览器控制台转义：

```javascript
decodeURIComponent('...')
```

## 案例3、邮件审批通知发送失败

页面显示邮件审批通知发送出去，但是查看incident日志失败。

检查项：
1. 策略通知是否配置了邮件服务器
2. 邮件服务器那边是否加了ucss的授权
3. 日志检查：`/opt/skyguard/sps/logs/sps-incident/incident.log`

常见错误：`501 Mail from address must be same as authorized`

解决方案：邮件通知里面配置的发件人地址必须和邮件服务器授权一致

## 案例4、邮件审批不发通知定位

日志显示 `restart next time` 但是一直没有执行，问题大概率是通知线程异常。

如果日志为 `notification is null` 说明没有配置策略通知。

## 案例7、webmail邮件方式审核点击“同意/拒绝释放”无法释放邮件

问题：用户通过webmail收到审批通知，点击“同意释放”无法释放邮件。

解决方案：通过邮件回复方式可以正常释放，webmail方式需要检查链接配置。
