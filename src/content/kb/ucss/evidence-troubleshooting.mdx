# UCSS证据故障排查

## 证据文件解密方法

### 分布式证据解密

分布式证据文件的解密和下载是由app负责的。

使用 `encfile` 工具：

```bash
root@UCSS3:/var/skyguard/forensics/encrypted/
./encfile
Usage of ./encfile:
  -d    Debug模式，打印Trace级别日志信息
  -e    是否加密
  -f string  指定要解析的文件名
  -m string  解析使用的算法: blowfish/sm4
  -t    探测加密算法
```

- `-f` 是解密，`-e` 是加密

解密示例：

```bash
./encfile -f 89af5697-116e-4033-9313-36158068...
```

### 集中式证据解密

这个工具在3.3的环境没有，拷贝到3.3的环境后可解密集中式证据。

```bash
root@UCSS3:/opt/skyguard/download#
java -Xms1024m -Xmx2048m -jar decryptForensics.jar xxx.eml
```

解密工具路径：

```bash
/opt/skyguard/sps/tools/decryptForensics/  #解密工具
decryptForensics.jar  readme.txt

cat readme.txt  #查看解密方法
```

## 事件正常，证据不存在

### 排查思路

**第一步：** 确定UCSS平台能查询到证据事件，获取流量uuid。

**第二步：** 进入证据目录查询对应的证据文件夹是否有该证据文件。

### 正常证据文件入库查询

根据网络事件uuid进行搜索：

```bash
ls /var/skyguard/sps/forensics/incident/network/<uuid>
# 例如: d84b2236-bf7c-4570-ae65-723c8aaf4e8f.eml
```

同时查看incident.log的证据文件入库状态：

```bash
grep <uuid> /opt/skyguard/sps/logs/sps-incident/incident.log
# 正常应显示: Incident forensics file received from dlp server
```

> 注意：如果grep uuid显示的是Incident forensics file，且证据文件后缀为.eml，如果未grep到则可能是被删除。

### 证据被删除的原因

证据文件被删除的原因是监控磁盘空间的minitor服务删除了证据文件。minitor巡检周期为1个小时一次。

> 证据空间回滚的最小粒度是天为单位，空间满后客户端只能显示当天的证据文件。

## 事件不存在

### 案例：网络事件不存在

查看incident.log日志，发现1个月前就已经没有incident日志了。

检查incident.error.log日志，发现 `Too many open files` 错误。

**解决方法：** 设置open files文件数限制。

## 证据空间遍历周期

证据空间遍历周期为1小时，minitor服务会定期检查证据存储空间，当空间不足时会按时间顺序删除最旧的证据文件。

## HTTP通道命中的证据问题

HTTP通道命中的证据文件可能存在格式或传输问题，需要检查证据上传通道状态和文件完整性。
