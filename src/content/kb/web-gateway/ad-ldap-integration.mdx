## Overview

Integrating with Active Directory (AD) or LDAP enables the WSG to:
- Apply web access policies based on user identity (not just IP address)
- Map users to AD groups for group-based policy assignment
- Provide single sign-on (SSO) experience via Kerberos/NTLM
- Generate user-attributed logs and reports

---

## Supported Authentication Methods

| Method | Description | SSO | Best For |
|--------|-------------|-----|----------|
| **NTLM** | Windows challenge-response | Yes | Windows domain environments |
| **Kerberos** | Ticket-based SSO | Yes | Domain environments |
| **LDAP Basic Auth** | Username/password prompt | No | Mixed environments |
| **IP-to-User Mapping** | DC agent maps IPs to users | Yes | Transparent auth |

---

## Step 1: Configure LDAP Server

1. Navigate to **Authentication > LDAP Servers**
2. Click **Add LDAP Server**
3. Configure the following:

| Field | Value |
|-------|-------|
| Server Type | Microsoft Active Directory |
| Host | `dc01.yourdomain.com` (or IP) |
| Port | 389 (LDAP) or 636 (LDAPS) |
| Base DN | `DC=yourdomain,DC=com` |
| Bind DN | `CN=ldap-svc,OU=ServiceAccts,DC=yourdomain,DC=com` |
| Bind Password | Service account password |
| User Attribute | `sAMAccountName` |
| Group Attribute | `memberOf` |

4. Click **Test Connection** to verify
5. Click **Save**

---

## Step 2: Configure Authentication Realm

1. Navigate to **Authentication > Realms**
2. Click **Add Realm**
3. Select the LDAP server configured above
4. Choose authentication method:
   - **NTLM / Kerberos** for transparent SSO
   - **Basic (LDAP Bind)** for direct credential validation
5. Set the realm name (e.g., `YOURDOMAIN`)
6. Click **Save**

---

## Step 3: Configure Kerberos SSO (Optional)

For transparent Kerberos authentication:

1. Join the WSG to the Active Directory domain (if applicable)
2. Create a service account with SPN:
   ```
   setspn -A HTTP/wsg.yourdomain.com WSG_SERVICE_ACCOUNT
   ```
3. Export the keytab file and upload to WSG:
   - Navigate to **Authentication > Kerberos**
   - Upload the keytab file
4. Configure DNS so clients resolve `wsg.yourdomain.com` to WSG IP

---

## Step 4: Import AD Groups

1. Navigate to **Authentication > Group Sync**
2. Click **Sync Now** to import AD groups
3. Groups will appear in the policy editor for assignment
4. Schedule regular sync (recommended: every 30 minutes)

---

## Step 5: Apply Group-Based Policy

1. Navigate to **Policy > Web Access Policy**
2. Create a policy:
   - **Users/Groups:** Select specific AD groups
   - **Action:** Allow / Block / Monitor
   - **Categories:** Select URL categories
3. Policy priority order: higher priority rules evaluated first
4. Click **Save & Apply**

**Example policy structure:**

| Priority | Group | Action | Categories |
|----------|-------|--------|------------|
| 1 | IT_Admins | Allow | All |
| 2 | Employees | Block | Gambling, Adult |
| 3 | Employees | Allow | All others |
| 99 | Default | Monitor | All |

---

## Step 6: Test Authentication

1. From a domain-joined client, browse to any website
2. With NTLM/Kerberos: No login prompt should appear
3. Verify user identity in **Logs > Web Logs** (should show username, not IP)
4. Navigate to **Reports > User Activity** to confirm user-level reporting

---

## Troubleshooting

| Issue | Possible Cause | Solution |
|-------|---------------|----------|
| LDAP bind failed | Wrong credentials or hostname | Verify bind DN and password |
| Users prompted for password | Kerberos not configured | Configure NTLM or Kerberos |
| Groups not appearing | Sync not run | Click "Sync Now" |
| Wrong user in logs | IP mapping issue | Check DC agent configuration |
| LDAPS connection failed | Certificate not trusted | Import DC SSL certificate |
