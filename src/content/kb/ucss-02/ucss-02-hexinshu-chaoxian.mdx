# 核心数超限案例

## 问题描述

客户反馈icap端口未监听，发现icap服务未启动，但cae相关服务启动正常，license也未过期。

> 说明：海光服务器出厂5100就是40c，有此问题打最新app补丁即可，做特批。

## 排查过程

### 1、查看icap日志

```bash
cat /opt/skyguard/ucs/var/log/IcapController.log
```

发现license status from 0 to 0，正常环境是1to1，怀疑是license过期，但license并未过期，其他服务器启动都正常。

### 2、查看dlpserver.log关于icap的日志

```bash
grep 'license ok' /opt/skyguard/ucs/var/log/dlpserver.log
```

更加怀疑是license的问题，status=3是什么意思呢，为什么之后icap会收到限制吗

### 3、查看dsg上获取的license信息

```bash
curl -s http://localhost:8888/app/v1/device/getLicense | python -m json.tool
```

发现 `licenseStatus: "3"`, `overload: true`

### 4、查看cpu核心数

```bash
lscpu
```

发现是56C，基本判断是核心数超限导致。

## 原因分析

License是5100，5100的限制是40C，所以dsg获取的license信息里面有licenseStatus: 3。

刚好，icap服务的启动，严格的校验了核心数，导致icap服务无法启动。

## 解决办法

重新申请11000的license，11000的license上限是64c（最新的已经给建行支持到128c了），我们出产标配是40c。
