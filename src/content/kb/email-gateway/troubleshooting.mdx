# Troubleshooting

This guide covers the most common issues encountered with the NextGuard Email Security Gateway and provides step-by-step resolution procedures.

---

## Email Not Received by Recipient

### Symptoms
- User reports they are not receiving email from an expected sender
- External sender reports their email was not delivered

### Diagnostic Steps

1. **Check Mail Trace:**
   - Navigate to **Email Gateway > Mail Trace**
   - Search by sender email and recipient email
   - Check the **Status** column for the message

2. **Common Causes and Solutions:**

| Status | Cause | Solution |
|--------|-------|----------|
| Quarantined (Spam) | Spam score exceeded threshold | Release from quarantine; add sender to allow list |
| Quarantined (Virus) | AV engine detected malware | Verify file is safe; override if false positive |
| Rejected (SPF Fail) | Sender IP not in SPF record | Contact sender to update their SPF record |
| Rejected (RBL) | Sender IP is blacklisted | Sender should request delisting from RBL |
| Deferred | Downstream server unreachable | Check internal mail server connectivity |
| Not Found | Message never reached gateway | Check MX record; verify external DNS |

3. **If message is not in Mail Trace:**
   - Verify MX records point to NextGuard gateway: `nslookup -type=MX yourdomain.com`
   - Check DNS propagation if MX was recently changed
   - Verify the sender actually sent the email (check their sent items)

## Legitimate Email Marked as Spam (False Positive)

### Symptoms
- Legitimate business email is quarantined or tagged as spam
- Recurring false positives from a specific sender or domain

### Resolution

1. **Immediate Fix:**
   - Navigate to **Quarantine**, find the message, click **Release**
   - Select **Release and Allow Sender** to whitelist

2. **Add to Allow List:**
   - Navigate to **Policy > Anti-Spam > Allow List**
   - Add sender email address or domain
   - Click **Save**

3. **Adjust Spam Score Threshold:**
   - If false positives are frequent, consider raising the quarantine threshold
   - Navigate to **Policy > Anti-Spam > Scoring**
   - Increase quarantine threshold from 70 to 75 or 80

4. **Train Bayesian Engine:**
   - Navigate to **Policy > Anti-Spam > Training**
   - Submit the false positive to retrain the machine learning model

5. **Check RBL Status:**
   - If the sender's IP is on a blocklist, they should request delisting
   - Use `mxtoolbox.com` to check IP reputation

## Spam Getting Through (False Negative)

### Symptoms
- Spam emails are reaching user inboxes
- Phishing emails bypass detection

### Resolution

1. **Review Spam Score:**
   - Use **Mail Trace** to find the message
   - Check the spam score breakdown
   - Identify which layers passed vs. missed

2. **Tighten Spam Thresholds:**
   - Navigate to **Policy > Anti-Spam > Scoring**
   - Lower the tag threshold to 30 (from 40)
   - Lower the quarantine threshold to 60 (from 70)

3. **Enable Additional RBLs:**
   - Navigate to **Policy > Anti-Spam > RBL Settings**
   - Ensure all recommended RBLs are enabled

4. **Block Sender/Domain:**
   - Navigate to **Policy > Anti-Spam > Block List**
   - Add the offending sender or domain

5. **Report for Training:**
   - Submit the missed spam to the Bayesian training engine
   - Navigate to **Policy > Anti-Spam > Training > Report False Negative**

## Outbound Email Rejected by External Server

### Symptoms
- Outbound email bounces with 550 or 553 error
- Receiving server rejects with "SPF failure" or "DMARC policy" message

### Resolution

1. **Check SPF Record:**
   ```bash
   nslookup -type=TXT yourdomain.com
   ```
   - Verify the gateway's outbound IP is included
   - Ensure SPF record does not exceed 10 DNS lookups

2. **Check DKIM Signing:**
   - Navigate to **Policy > Anti-Spoofing > DKIM > Signing**
   - Verify DKIM signing is enabled and key is valid
   - Test with `mail-tester.com` or `dkimvalidator.com`

3. **Check DMARC Record:**
   ```bash
   nslookup -type=TXT _dmarc.yourdomain.com
   ```
   - Verify alignment mode matches your mail flow

4. **Check IP Reputation:**
   - Verify outbound gateway IP is not blacklisted
   - Check at `mxtoolbox.com/blacklists.aspx`
   - If listed, submit delisting request to the RBL provider

5. **Check Reverse DNS (PTR Record):**
   ```bash
   nslookup <gateway-outbound-IP>
   ```
   - PTR must resolve to a hostname matching your domain
   - Contact your ISP if PTR record is missing

## Gateway Performance Issues

### Symptoms
- Email delivery delays
- High queue buildup
- Admin console slow to respond

### Diagnostic Steps

1. **Check System Health:**
   - Navigate to **Dashboard > System Health**
   - Monitor:
     - **CPU usage:** Should be < 70% sustained
     - **Memory usage:** Should be < 80%
     - **Disk usage:** Should be < 85% (log files may fill disk)
     - **Queue size:** Should process and reduce over time

2. **Check Mail Queue:**
   - Navigate to **Email Gateway > Queue**
   - Large queue indicates delivery issues
   - Check if messages are deferred (downstream server issue)

3. **Common Fixes:**

| Issue | Solution |
|-------|----------|
| High CPU | Reduce concurrent scan threads; disable unused scanning features |
| High memory | Restart gateway service; check for memory leaks in logs |
| Disk full | Archive or delete old logs; increase disk space |
| Large queue | Check downstream mail server; verify DNS resolution |
| Slow admin console | Clear browser cache; restart admin web service |

4. **Restart Services:**
   ```bash
   systemctl restart nextguard-mta
   systemctl restart nextguard-admin
   ```

## MX Record and DNS Issues

### Verifying MX Records

```bash
nslookup -type=MX yourdomain.com
```

Expected output should show your NextGuard gateway:
```
yourdomain.com  MX  10  mail.yourdomain.com
```

### Common DNS Issues

| Issue | Symptom | Solution |
|-------|---------|----------|
| Missing MX record | No email reaches gateway | Add MX record to DNS |
| Wrong MX priority | Email goes to wrong server | Set correct MX priority |
| Stale DNS cache | Recent MX change not working | Wait for TTL expiry (or flush DNS) |
| PTR record mismatch | Outbound email rejected | Update PTR via ISP |

### Testing DNS Resolution from Gateway

```bash
dig MX yourdomain.com
dig A mail.yourdomain.com
dig TXT yourdomain.com
```

## TLS / SSL Certificate Issues

### Symptoms
- STARTTLS failures in mail log
- Certificate warning errors
- Outbound delivery fails with TLS-required servers

### Resolution

1. **Check Certificate Validity:**
   - Navigate to **Settings > Certificates**
   - Verify certificate is not expired
   - Verify certificate chain is complete (intermediate CA included)

2. **Renew/Replace Certificate:**
   - Generate a new CSR if certificate is expiring
   - Upload new certificate via **Settings > Certificates > Upload**
   - Include private key and full certificate chain

3. **TLS Configuration:**
   - Navigate to **Settings > TLS**
   - Recommended: **Opportunistic TLS** (use TLS when available)
   - For compliance: **Mandatory TLS** for specific partner domains
   - Minimum protocol: TLS 1.2

## Quarantine Not Working as Expected

### Messages Not Being Quarantined

1. **Check policy thresholds:**
   - Navigate to **Policy > Anti-Spam > Scoring**
   - Verify quarantine threshold is set correctly
   - Ensure the policy action is set to **Quarantine** (not Tag)

2. **Check allow list:**
   - Navigate to **Policy > Anti-Spam > Allow List**
   - Verify the sender is not on the allow list

### Users Cannot See Quarantine

1. **Enable user self-service:**
   - Navigate to **Policy > Anti-Spam > End-User Settings**
   - Ensure **User Self-Service Quarantine** is enabled

2. **Check quarantine digest:**
   - Navigate to **Policy > Anti-Spam > Quarantine Digest**
   - Verify digest notifications are enabled
   - Verify recipient email addresses are correct

## HA (High Availability) Issues

### Failover Not Triggering

1. **Check heartbeat:**
   - Navigate to **Settings > HA > Status**
   - Verify heartbeat between primary and secondary nodes
   - Check network connectivity between nodes

2. **Manual failover test:**
   - Navigate to **Settings > HA > Manual Failover**
   - Test failover to verify secondary takes over correctly

3. **Sync issues:**
   - Verify configuration sync between primary and secondary
   - Check sync logs: **Logs > System Log > HA Sync**

## Collecting Diagnostic Information

When contacting NextGuard Support, collect the following:

1. **System Information:**
   - Navigate to **Settings > System Info**
   - Note: firmware version, serial number, license status

2. **Export Diagnostic Package:**
   - Navigate to **Settings > Diagnostics > Export**
   - Click **Generate Diagnostic Package**
   - This creates a bundle including: system logs, configuration, mail queue status, health metrics

3. **Relevant Log Entries:**
   - Export logs from the time period of the issue
   - Include Mail Log, System Log, and any threat logs

4. **Mail Trace Results:**
   - Export the mail trace for affected messages
   - Include full SMTP transaction details
